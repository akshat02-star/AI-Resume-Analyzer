backend

1. DONE: parse_pdf is a synchronous blocking call inside an async route
PdfReader is synchronous. Calling it directly in an async route blocks the event loop — no other requests can be handled while a PDF is being parsed. Industry fix: wrap it in asyncio.run_in_executor to offload to a thread pool.

2. TODO: No file size limit
A user could upload a 500MB PDF and it would be accepted. Industry standard is to enforce a max file size (e.g. 10MB for resumes) before doing any parsing.

3. TODO: No error handling around PDF parsing
PdfReader can raise exceptions for corrupted, password-protected, or malformed PDFs. If it throws, FastAPI will return a generic 500 with no useful message. You should catch those and return a proper 400/422.

4. DONE: parse_pdf lives in analyzer.py — wrong separation of concerns
Parsing is an I/O utility concern; analysis is a business logic concern. Industry practice keeps these in separate modules (e.g. utils/pdf.py for parsing, services/analyzer.py for Gemini logic).

5. TODO: analyze is async but does no async work
Once the real Gemini call is added, this is fine. But currently it's misleadingly async. More importantly — the Gemini client call should use the async SDK (google-generativeai has async support) otherwise you have the same blocking problem as #1.

6. TODO: No response model defined
FastAPI supports response_model on routes. Without it, there's no documented contract for what the API returns — important for frontend teams and API documentation (Swagger).

7. TODO: No file content sanitization / empty text guard
A valid PDF can have zero extractable text (e.g. scanned image-only PDF). parse_pdf would return an empty string and that would get passed to Gemini silently. Should check if text is empty and return a clear error.

8. TODO: using NGinx reverse proxy
The most critical ones to fix before any real usage: #1, #3, and #7.
